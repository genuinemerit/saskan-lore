# NFR-005: Pre-Pilot Pragmatism

## Requirement

The design may favor simplicity and speed of learning over long-term elegance, provided all artifacts remain inspectable and reusable.

## Rationale

This is explicitly a learning and validation exercise. Over-engineering at this stage wastes time without confirming that the core pipeline works. See STATUS.md and ADR-006 (graduation criteria).

## Implications

- Skipping abstractions that are not yet needed is acceptable.
- Skipping documentation, leaving the schema undocumented, or producing data in an uninspectable state is not acceptable.
- Code must be readable and runnable, even if not fully generalized.
- Decisions that trade elegance for speed must still be recorded (as ADRs if significant, as inline comments if minor).
- "Throw-away" does not mean undocumented or opaque — it means not over-built.
